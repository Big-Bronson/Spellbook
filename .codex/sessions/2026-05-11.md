### Key Decisions

The session codified patterns that were already in use but hadn't been explicitly documented. The ADRs capture trade-offs around safety (SecureString for passwords, return vs exit in public scripts), compatibility (minimum version pinning on RequiredModules, portable CSPRNG), and operational resilience (per-iteration logging to catch partial failures, defensive disconnect guards). The dispatch model via dot-sourcing was formalized as the core architectural choice enabling persistent connections across an interactive toolkit.

### Lessons and Gotchas

The ADR collection reveals that decisions which look arbitrary in code (why SecureString instead of plaintext? why return instead of exit?) have non-trivial answers rooted in safety, portability, and operator experience. The dispatch model's reliance on dot-sourcing is elegant but requires scripts to be defensive about connection state — they must check if a connection exists before attempting auth, or risk failures. The decision to document architecture decisions after the fact (ADRs 0001-0018 were backfilled) underscores the importance of the new CLAUDE.md guidance: future sessions should flag undocumented decisions immediately so the "why" doesn't get lost.
